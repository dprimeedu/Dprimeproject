## 1. 변형문제 HWPX 페이지 밀도 튜닝 ⏳ **진행 중 (결정 대기)**

### 증상
모의고사 변형문제 HWPX(`[프라임에듀]_..._변형문제.hwpx`)에서 한 페이지(2단)에
순서 문제가 **한 단에 1개씩(2개/page)**만 들어가고 아래 절반이 비었음.

### 원인 (실측 확인)
- 파일: [common/hwpx/hwpx_builder.py](../common/hwpx/hwpx_builder.py) — `_ColumnTracker`가
  문제 개수가 아니라 **예상 높이(줄 수)**로 단/페이지를 나눔. 문제를 경계에서 안 쪼갬.
- 순서 문제 1개 = 추정 **23~28줄**(지문 박스가 16~21줄). 한 단 = 약 49줄.
  두 문제 합 ≈52줄 > 49 → 한 단에 1개씩만 들어감.
- 양식 줄간격이 **발문/선택지 160% · 지문박스 150%**로 커서 단 용량이 작았던 게 핵심.

### 지금까지 한 것
- `COL_FILL_TOLERANCE`(단 채움 허용오차) 추가 → 효과 작아 한계 확인.
- **줄간격을 `LINE_SPACING_PCT`(현재 135%)로 통일** (빌드 시 paraPr 0/1/13/14에 적용).
  한 단 용량 49→약 55줄로 늘어 문제 2개가 안전하게 들어감. 추정도 같은 값으로 맞춤(쪼개짐 없음).
- 배포 완료(commit `b6501d81`).

### ★ 조절 방법 (상수 하나)
[hwpx_builder.py](../common/hwpx/hwpx_builder.py)의 `LINE_SPACING_PCT` 값만 바꾸면 됨:
- `150` = 양식 기본(헐거움, 원래 상태)
- `135` = 현재값(빽빽, 2개/단)
- `140`~`145` = 중간(덜 답답, 가끔 1개/단으로 돌아감)
- 바꾼 뒤 `git push` 하면 1분 내 자동 반영(아래 2번 참고).

### 미해결 결정 (사용자 피드백 대기)
사용자가 "줄간격 줄이면 답답해 보인다"고 함. 135% 출력 보고 판단 중. 선택지:
1. 135% 유지 / 140~145%로 완화 / 130%로 더 빽빽 — `LINE_SPACING_PCT`만 조정.
2. **대안(미구현): 줄간격은 150% 그대로 두고 페이지 여백을 줄여** 단을 길게.
   글자 간격 안 건드려 답답함 없음. 양식 `section0.xml`의 margin
   (`header="4252" footer="4252" top/bottom="2834"`)을 빌드 시 줄이고
   `MARGIN_HEADER/FOOTER/TB` 상수도 같이 맞춰야 함(추정 일치). header/footer 4252→2126 정도면 단 +3~4줄.
   → 사용자가 "여백 방식" 원하면 이걸로 구현.

## 2. 자동배포 시스템 (다른 PC에서 push 전 꼭 숙지) ✅

**push 하면 1분 내 운영 자동 반영됨.** 단, 구조가 한번 깨진 적 있어 정리해 둠.

- **운영:** NAS Docker 컨테이너 `django`, 브랜치 `main`. URL https://dprimeedu.synology.me
- **자동배포 주체:** NAS **DSM 작업 스케줄러**의 작업 **"Dprimeauto"**(root, 매분)
  → `/bin/sh /volume1/homes/admin/deploy-watch.sh` 실행.
  이 스크립트가 `docker exec`로 fetch→pull→collectstatic→gunicorn 재기동.
  (예전 컨테이너 내부 cron 은 컨테이너 재생성 시 유실돼서 제거하고 호스트 방식으로 바꿈.)
- **검증:** push 후 `https://dprimeedu.synology.me` 200 뜨고, 원하면 NAS에서 last-deployed 확인.

### ⚠️ 함정
- **배포가 안 될 때 제일 먼저 의심:** stale lock.
  → `ssh dprime-nas 'sudo /usr/local/bin/docker exec django rmdir /tmp/auto-deploy.lock'`
- gunicorn 재기동은 반드시 `docker exec -d`(또는 컨테이너 내부 setsid). 절대
  `docker exec`(blocking)+`setsid &` 쓰지 말 것 — exec 종료 시 자식 같이 죽어 502.
- ⚠️ **다른 PC에는 NAS SSH 키(`~/.ssh/dprime_nas`)가 없을 수 있음.** SSH 작업 필요하면
  키 복사하거나 DSM에서 직접. 단순 코드 작업은 `git push`만으로 배포되니 SSH 불필요.

### 수동 배포(필요 시, NAS에서)
```bash
ssh dprime-nas 'sudo /usr/local/bin/docker exec django bash -c "cd /home/Dprimeproject && git pull --ff-only origin main"'
# gunicorn 재기동
ssh dprime-nas 'sudo /usr/local/bin/docker exec django bash -c "pkill -KILL -f \"venv/bin/python3 .*gunicorn\"; true"'
ssh dprime-nas 'sudo /usr/local/bin/docker exec -d -w /home/Dprimeproject django bash -c "/home/Dprimeproject/venv/bin/gunicorn --bind 0.0.0.0:8000 --workers 3 --threads 2 --worker-class gthread --timeout 120 config.wsgi:application >> /var/log/gunicorn.log 2>&1"'
```
