## 1. 환경 통일 — Postgres에서 SQLite로

### Why
- 학원 규모(동시 사용자 수십명)에선 SQLite + WAL 충분
- 백업 단순화 (`db.sqlite3` 파일 1개 복사로 끝)
- 협업자 환경 일치 (협업자도 SQLite 사용 중이었음)
- NAS 운영 부담 감소 (Postgres 서비스 24/7 관리 불요)

### 변경 결과
- `config/settings_dump.py` 삭제 (Postgres 전용 오버라이드였음)
- 기본 `config/settings.py`는 원래부터 SQLite 설정 — 그대로 사용
- NAS도 이미 SQLite로 운영 중이었음 (5/16 진행상황 doc에 적힌 Postgres는 outdated였음)

## 2. 모의고사 sync 시스템 (Phase 1 + 2)

### 핵심 명령

```powershell
conda activate project
cd "z:\home\Drive\문서 자동화 작업\Workplace\Dprimeproject"
python manage.py sync_exam_data
```

### 무엇이 동작하는가

1. **입력**: `Z:\home\Drive\문서 자동화 작업\엑셀자료모음\모의고사 EXCEL\` 의 4개 엑셀 파일
   - `고1 전체 모의고사 DB.xlsx`
   - `고2 전체 모의고사 DB.xlsx`
   - `고3 전체 모의고사 DB.xlsx`
   - `부교재_통합엑셀_목록.xlsx` (이건 메타데이터라 sync 대상 아님)

2. **처리**: 각 파일의 14-15개 시트를 → 16개 SQLite 테이블에 매핑
   - 시트 종류: 문제출력 / 내신단어 / 상세해설 / 어법1·2·3단계 / 중요영작 / 요약문완성 / 원문추가 / 내신TEST / 서술형 / 내신빨파 / 객관식빈칸 / 원문 / 보기변형(고3만)
   - 자연키 매칭 — 같은 (학년/연도/강/번호/유형) 조합은 추가 안 함
   - 단원 컬럼('2019-03-18' 형식)에서 연도/월/번호 폴백 파싱
   - 컬럼 alias (고2 원문은 '지문', 원문추가는 '년' 같은 변종 대응)

3. **출력**: 추가/수정/변경없음/오류 카운트 + 시트별 세부 결과
   - SyncLog 모델에 자동 기록됨
   - `/admin/academy/synclog/` 에서 웹UI로 확인

### 옵션

| 옵션 | 의미 |
|---|---|
| `--dry-run` | 실제 변경 없이 미리보기만 |
| `--grade 고1` | 한 학년만 처리 |
| `--sheet 내신단어` | 특정 시트만 처리 |
| `--limit 100` | 시트당 첫 N행만 (디버깅용) |
| `--path /외부경로` | Excel 폴더 다른 곳에 있을 때 |
| `--triggered-by web` | 트리거 기록 구분 (cli/web/auto) |

### 안전 장치

- ✅ **Idempotent**: 같은 명령 여러 번 돌려도 데이터 망가지지 않음
- ✅ **트랜잭션**: 실 적용 시 전체 한 트랜잭션 — 도중 실패 시 자동 rollback
- ✅ **추가/수정만**: 삭제 정책 안 함 (엑셀에서 행 지워도 DB에 남음)
- ✅ **자연키 매칭**: 색인 충돌 회피 위해 max+1 자동 발급

### 감사 로그 (Phase 2)

- 모델: `academy.SyncLog`
- 기록: 시작·종료 시각 / 트리거 / dry-run 여부 / 학년·시트 / 추가·수정·변경없음·오류 / 시트별 세부 결과 / 오류 상세
- 웹UI: `/admin/academy/synclog/`
- 클릭하면 학년×시트 매트릭스 + 오류 상세 표시

## 5. 발견한 이슈 / 향후 정리할 것

### 5-1. migrations 0001/0002 의존성 순서 문제

**현상**: `academy/migrations/0001_initial.py`이 detail 테이블(Grammarlv2_Data 등)을 만들면서 KeyTable에 FK를 거는데, KeyTable은 0002에서야 만들어짐 → "lazy reference 'academy.keytable'" 에러.

**임시 우회**: NAS에선 SyncLog 테이블을 raw SQL로 직접 만들고 `django_migrations` 테이블에 마크만 했음 (`migrate` 통과 못 함).

**근본 해결**: 마이그레이션 0001부터 다시 만들기. 의존성 순서대로:
1. KeyTable 먼저
2. 그 다음 KeyTable 참조하는 detail 테이블들

또는: KeyTable에 `managed = False` 추가하면 마이그레이션이 신경 안 씀. 그러나 이 경우 Django가 더 이상 KeyTable 관리 안 함.

지금은 우회로 잘 굴러가니까 시급하진 않음. 다음에 마이그레이션 손볼 때 정리.

### 5-2. docker-compose.yml과 실제 운영 컨테이너 불일치

`docker-compose.yml` 에 적힌 구성:
- `web` (django-gunicorn): python:3.10-slim 빌드

실제 NAS 운영:
- `django` (이름 다름): ubuntu:24.04 베이스
- 컨테이너 안에 자체 venv 만들어서 운영

문서가 현실과 다르면 다음 배포·재구축 시 혼란. 정리하면 좋음 (docker-compose.yml 업데이트 또는 README에 실제 구성 명시).

### 5-3. 백업 파일 정리

NAS에 `db.sqlite3.backup-20260518-214951` (139MB) 1주일쯤 두고 문제 없으면 삭제. 자동 정리 cron 같은 거 잡으면 더 좋음.
