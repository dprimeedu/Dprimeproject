## 3. POST 핸들러 — KeyTable 매칭 + 결과 미리보기

GET 빈 폼 → POST 매칭 → **같은 페이지 상단에 결과 카드 + 폼은 선택값 유지**.

### 매칭 규칙
- `unit_<grade>` → `KeyTable.month` (3월 = `'3'`)
- `qnum_<grade>` → `KeyTable.number` (zero-pad 변형 자동 포함, `'1'` ↔ `'01'`)
- 옵션이 detail 테이블이면 `pk_number__in=matched_pks`로 카운트
- `문제출력`만 `QuestionData`(학년/강/번호 직접 매칭)

### `_number_variants(values)`
```python
def _number_variants(values):
    out = set()
    for v in values:
        v = v.strip()
        if not v:
            continue
        out.add(v)
        if v.isdigit():
            out.add(v.zfill(2))
            out.add(v.lstrip('0') or '0')
    return out
```

### 결과 카드 구성
- 매칭된 KEY_TABLE 행 수 / 이름 / 시험구분
- 옵션별 detail row 수 (4열 그리드)
- 매칭 행 상위 30개 표 (교재/학년/연도/강/번호/색인)

### 검증
- 고1·3월·번호 18~20 → KeyTable 52건, RedBlue 80건, DetailedExpl 30건


어제(5/18) sync한 KeyTable은 모의고사 한정. EBS/교과서를 같이 적재하려면 카테고리·교재 구분이 필요.

```python
class KeyTable(models.Model):

        ('ebs_high3',  'EBS 고3'),
        ('ebs_naesin', 'EBS 내신교재'),

        db_table = 'KEY_TABLE'
```

## 7. sync_textbook_data 명령

`academy/management/commands/sync_textbook_data.py` 신규. 어제 만든 `sync_exam_data`의 detail 시트 로직(`SHEET_SPECS` + 매칭 로직)을 재사용하면서, **단원 형식만 자유 텍스트(`1-1`/`1-A`)로 분기**.

### 핵심 차이
| 항목 | sync_exam_data (모의고사) | sync_textbook_data (EBS/교과서) |

| 입력 | 학년별 단일 `고N 전체 모의고사 DB.xlsx` | 카테고리 폴더 walk `*통합*.xlsx` |

| 책 식별자 | 없음 (모의고사는 강+번호로 unique) | `book` 컬럼 — 파일명에서 추출 |

- `--category {mock_exam,ebs_high3,ebs_naesin,textbook,all}`

- `*통합*.xlsx` (락파일 `~$` 제외)

## 8. SHEET_SPECS 확장 — 14개 시트 매칭

처음 EBS 통합본 sync 시 매칭되는 시트는 RedBlue + DetailedExpl 2개뿐. 컬럼명/시트명이 모의고사 통합본과 미세하게 달랐기 때문.

### 도입한 두 가지 메커니즘

**(1) `optional_cols`** — 부재해도 시트 처리 진행 (해당 컬럼은 빈 문자열로 채움)
```python
"내신단어": {
    "table": "WordTest",
    "cols": {"word": "단어", "korean_definition": "해석"},
    "optional_cols": {
        "english_definition": "영영",  # EBS는 별도 '영영단어' 시트로 분리됨
    },
    "natural_key": ["word"],
},
```

**(2) Alias 리스트** — 컬럼명이 다른 통합본 호환
```python
"중요영작": {
    "cols": {
        "Sentence":    ["영작", "영어"],   # EBS는 '영어'
        "Translation": ["해석", "한글"],   # EBS는 '한글'
    },
    "optional_cols": {
        "ETC":          "회화/Extra/기타여부",
        "Key_sentence": "중요영작",
    },
    ...
},

"보기변형": {
    "cols": {
        ...
        "Modified": ["(변형)정답", "변형"],  # 모의고사 vs EBS
    },
    ...
},
```

### `SHEET_NAME_ALIASES`
시트명 자체가 다른 경우 — EBS는 `변형문제`, 모의고사 spec key는 `보기변형`:
```python
SHEET_NAME_ALIASES = {
    "변형문제": "보기변형",
}
```

### 적용 후 비교
| 통합본 | 적용 전 | 적용 후 |
|---|---|---|
| EBS 2024 수능완성 | 2개 시트 (+422건) | 동일 (해당 통합본은 변별유형이 빈 상태) |
| EBS 2026 올림포스1 전국연합 | — | **14개 시트 / +22,048건** |
| 모의고사 회귀 (`sync_exam_data --dry-run`) | — | **13개 시트 모두 "변경없음"** (안전) |

### 남은 이슈
보기변형 시트는 자연키가 `["Question"]` 단일이라 같은 PK 안에 동일 Question 다중 행이 있을 때 INSERT/UPDATE 반복됨 (2회차 sync에서 ~5851 update). 자연키를 `["Question", "Modified"]` 또는 `["Question", "Qtype"]`로 강화하면 해소 가능 — 기존 모의고사 데이터 영향 검토 후 후속.

## 9. 다운로드 4종 — CSV / Excel / PDF / HWP(보안)

`DownloadView` (`POST /tools/sub-book/download/`)에 `format` 파라미터로 4가지 분기:

| 버튼 | format | 응답 |
|---|---|---|
| CSV | `csv` | UTF-8 BOM + 7컬럼 (교재/학년/연도/강/번호/색인/카테고리) |
| Excel | `xlsx` | openpyxl Workbook |
| **PDF** | `pdf` | **WeasyPrint** HTML → PDF (A4 + 한글 폰트) |

JS가 폼 데이터를 hidden form으로 재구성해서 새 form submit — 현 페이지를 떠나지 않고 브라우저가 파일로 받음.

검증 (E2E):

## 10. .pedu 보안 포맷 설계

운영자가 정리해준 흐름:
```
HWP 다운로드 버튼
  ↓
서버는 평문 HWP가 아닌 '암호화된 데이터 파일'만 내려보냄
  ↓
사용자 PC의 변환 프로그램(exe)이 복호화 + 한컴오피스 호출 → HWP 생성
```

목적: 서버에는 win32com 안 두고(NAS Docker = Linux), HWP 변환은 사용자 PC의 한컴이 담당.

### 파일 레이아웃
```
+0   [4]   magic = b"PEDU"
+4   [1]   version = 1
+5   [12]  AES-GCM nonce
+17  [4]   user_id (uint32 BE)
+21  [8]   expiry_unix (uint64 BE)
+29  [32]  HMAC-SHA256(hmac_key, aad + ciphertext)
+61  [N]   AES-256-GCM(gzip(json(payload)))   # aad 로 헤더 묶음
```

### 검증 순서 (`decrypt_pedu` 5단계)
1. magic / version
2. expiry_unix > now (시계 슬립 60초 허용)
3. HMAC constant-time 비교
4. AES-GCM 자체 인증태그 검증
5. gzip 해제 + JSON 파싱

### Payload 스키마 `pedu/sub_book/v1`
```json
{
  "schema": "pedu/sub_book/v1",
  "generated_at": 1779190599,
  "name": "백현고1",
  "test_type": "모의고사",
  "dataset": "mock_exam",
  "grade_selection": {...},
  "options": ["red_blue", "detailed_expl"],
  "key_table": [{grade, year, month, number, total_number, book, category}, ...],
  "details": {"red_blue": [...], "detailed_expl": [...], ...}
}
```

### 키 운영
- `PEDU_AES_KEY`, `PEDU_HMAC_KEY` (둘 다 base64 32바이트)
- `config/settings.py:166` 에서 `.env`로부터 읽기, 부재 시 결정적 dev 키로 폴백 (운영 부적합)
- `secure_export.generate_master_keys()` 헬퍼로 운영 키 생성

### 단위테스트 (5건 모두 통과)
- round-trip (한글 포함 payload 무손실)
- HMAC 변조 거부 (페이로드 1바이트 flip → `PeduHmacError`)
- 매직 헤더 거부 → `PeduMagicError`
- 만료 거부 (`expiry_hours=-0.1`) → `PeduExpiredError`
- 키 생성 헬퍼 (base64 44바이트 × 2)

## 12. NAS 배포 — SSH + Docker exec 자동화

### 초기 권한 잡기
- admin 계정은 docker 그룹 없음 → sudo 필요
- sudo는 비밀번호 입력. PowerShell `Read-Host` → 임시 파일 `$env:TEMP\nas_pw` → SSH stdin으로 `sudo -S` 전달
- (도중에 비번 "1"로 잘못 알려준 해프닝 있었음 — 결국 11바이트 정상 비번으로 통과)

### 영구 권한 부여
```bash
sudo chgrp administrators /var/run/docker.sock
sudo chmod 660 /var/run/docker.sock
```
→ 이후 admin이 docker 명령을 sudo 없이 사용 가능. **단, NAS 재부팅 시 리셋**.

### 코드 동기화 — tar pipe + docker exec
NAS에 scp가 막혀 있었음 (sftp 비활성). 해결:
```bash
# 로컬에서
tar -czf /tmp/dprime_changes.tar.gz <변경된 9개 파일/폴더>

# pipe로 직접 컨테이너에 풀어넣기 (sudo 없이!)
cat /tmp/dprime_changes.tar.gz | ssh -i ~/.ssh/dprime_nas admin@192.168.0.100 \
    '/var/packages/Docker/target/usr/bin/docker exec -i django tar xzvf - -C /home/Dprimeproject/'
```

마운트가 `/volume1/Docker → /home`이라 컨테이너 내부에서 풀어도 호스트 파일에 즉시 반영됨.

### 패키지 설치 — apt + venv pip
```bash
# 시스템 의존성 (WeasyPrint용)
docker exec django apt-get install -y \
    libpango-1.0-0 libpangoft2-1.0-0 \
    libharfbuzz0b libharfbuzz-subset0 \
    libcairo2 \
    fonts-noto-cjk

# Python 패키지 (venv 사용!)
docker exec django /home/Dprimeproject/venv/bin/pip install \
    weasyprint==68.1 cryptography reportlab==4.5.1 \
    google-genai google-generativeai
```

### 마이그레이션 + reload
```bash
docker exec django python3 manage.py migrate academy
docker exec django bash -c 'kill -HUP <gunicorn_master_pid>'
```
HUP 시그널로 graceful reload (워커 PID 교체 확인).

### 최종 검증
컨테이너 안에서 Django Client로 `POST /tools/sub-book/download/ format=pdf`
→ **200 / application/pdf / 296,391 bytes / `%PDF-1.7` 매직** ✅

## 13. 발견한 운영 환경 차이

NAS 배포 과정에서 알게 된 사실들 — 다음 작업에 도움될 메모.

| 발견 | 내용 |
|---|---|
| **컨테이너 이미지** | docker-compose.yml에는 `build: .` (Dockerfile = python:3.10-slim) 적혀 있지만, 실제 실행 중인 컨테이너는 **ubuntu:24.04** + 수동 셋업 형태 |
| **gunicorn 위치** | `/home/Dprimeproject/venv/bin/gunicorn` — venv 격리. 패키지 추가는 항상 **venv pip** |
| **마운트** | 컨테이너 안 `/home` ↔ 호스트 `/volume1/Docker`. 코드는 `/home/Dprimeproject` |
| **scp/sftp** | 비활성. ssh stdin + `docker exec -i tar xzvf -`로 우회 가능 |
| **docker 바이너리** | `/var/packages/Docker/target/usr/bin/docker` (PATH에 없음, 절대경로 호출 필요) |
| **admin 권한** | administrators 그룹 (101). docker 그룹 미존재. 소켓 권한으로 administrators에 부여 |

이 차이 때문에 **Dockerfile / docker-compose.yml은 현 운영과 형상이 어긋난 상태**. 컨테이너를 다시 만들면 이번 apt + pip 작업이 모두 날아감.
