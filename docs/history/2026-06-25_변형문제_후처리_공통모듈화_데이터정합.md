## B. 2026 모의고사 5회차 로컬↔서버 데이터 정합

### B-1. 발견된 큰 함정

처음에 ORM 으로 변형문제 수를 셌더니 한 회차당 18~25 정도가 나왔다.
"한 회차에 변형이 25개라고? 최소 300은 돼야 하는데" 라는 지적으로 다시 확인.

원인: `ModifiedQuestions_Data` 모델이 `OneToOneField(KeyTable, primary_key=True)`
로 선언되어 있지만, 실제 DB 는 `managed=False` 라 OneToOne 제약을 강제하지 않고
1:N 데이터가 들어가 있음. Django ORM `set(values_list("pk_number", flat=True))`
같은 카운팅은 distinct pk_number 만 잡아서 실제의 1/7 정도로 줄어듦.

**교훈**: `managed=False` 테이블에서 변형문제 행 수 셀 때는 raw SQL 또는
`.filter(pk_number__in=...).count()` 처럼 명시적 카운트로 가야 함.

정정한 회차별 변형문제 수:

| 회차 | 변형 |
|---|---:|
| 2026 고1 3월 | 363 |
| 2026 고1 6월 | 359 |
| 2026 고2 3월 | 267 |
| 2026 고2 6월 | 359 |
| 2026 고3 3월 | 341 |
| 전체 DB | **56,955** (KeyTable 7,870 × 평균 7.2) |

### B-3. 정리 흐름

**① 고2 6월 서버에서 9개 삭제** — 자연키(total_number + qtype + 지문 첫 80자)
매칭으로 server-only 행을 정확히 식별. 다중 매칭 0건, 단일 매칭 9건. rowid 기반
DELETE 로 안전하게 제거.

**② 고1 3월 / 고2 3월 의 "로컬이 더 많음" 처리** — 처음엔 서버에 업로드하려고
KEY_TABLE 슬롯 매핑까지 준비했지만, 빠진 슬롯이 18·19·26·43 같은 번호로 나옴.
원장님 확인 결과 "쓸데없는 번호" — 어디서 잘못 끼어든 색인. 그래서 업로드 대신
**로컬에서 그 단원 코드의 행 모두 삭제** 로 가닥.

삭제 결과:
- 고1 3월: 단원 2026-03-{18,19,26,43} → 46행 삭제 (407 → 361)
- 고2 3월: 단원 2026-03-{27,28,43} → 3행 삭제 (270 → 267)
- 백업 xlsx 옆에 보관 (`...쓸데없는번호삭제전백업_*.xlsx`)

**③ 고1 3월 잔존 orphan 2개** — ②에서 단원 19/43 행이 빠지면서 서버에 그
번호의 변형 2개(어법·어휘, 후처리 None 케이스)가 고아로 남음. 같은 자연키 매칭으로
